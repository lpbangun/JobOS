Close JobOS’s application-evidence gap without adding an external action. The loss to minimize is any state in which local approval, an application status, or an agent claim can be mistaken for proof of what was actually submitted.

A trusted local user must be able to freeze an immutable application packet for an approved job/profile. The packet must identify the exact approved resume and optional approved cover revisions by artifact ID and SHA-256, the exact redacted answer row versions, target identity, proof IDs, score snapshot, blockers, and warnings. Equal inputs hash equally. Changes to materials, answers, or target identity make the old packet historical/non-attestable; the next explicit freeze creates a new revision or attempt without erasing prior evidence. Do not auto-create packets from pursue, review, or redrafting.

A trusted local user must then be able to attest submission against one exact current packet and optionally confirm an external reference. Attestation is local user evidence, never a JobOS-performed submission: every response and audit must state no external side effect and `submissionPerformed: false`. A pre-apply tracking status may advance to `applied` only while the receipt and exact packet-bound status history commit atomically. Later statuses never regress. Exact receipt replay is idempotent; conflicting immutable evidence fails typed and never overwrites history. Bare manual `applied` tracking remains backward compatible but creates no receipt and is explicitly `receiptBound: false`.

No answer plaintext—public, personal, sensitive, or restricted—may enter packet/receipt storage, hashes derived from secret values, audit payloads, CLI/agent/TUI output, diffs, or workspace mirrors. Pin answer identity with non-secret row-version fingerprints. SQLite is canonical; packet/readiness/job YAML and audit JSONL appear only after successful guarded persistence. A stale or losing writer must leave no half packet, receipt claim, status change, or projection.
