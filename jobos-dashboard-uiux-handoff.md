# JobOS Minimal Web Dashboard — UI/UX Handoff

**Audience:** implementation owner building the single local HTML dashboard for MVP 1 (web dashboard + SQLite/Postgres).
**Scope:** minimal web dashboard and UX flow for technical local users.
**Explicit non-goals:** polished consumer SaaS visuals, browser extension, external sending/applying, mobile-native apps, auth/tenancy, analytics-heavy design system.

---

## 1. North Star for the Dashboard

The dashboard is a **local control panel for the same canonical state the CLI, REST API, and MCP tools read and write**. It is not a separate product surface with its own data model or special-case actions. Every screen, button, and status must be implementable as a thin view over the canonical entities defined in `ideal-agent-native-job-application-app.md`:

`Profile`, `ResumeSource`, `Job`, `Company`, `Stakeholder`, `Application`, `Artifact`, `Task`, `OutreachThread`, `Offer`, `AutomationRun`.

Key UX values for the MVP dashboard:

1. **Honest about agent assistance.** Show when an artifact, score, or task was generated by an agent/model, by which command/run, and its approval state.
2. **No hidden external actions.** Buttons that could touch the outside world (send message, submit application) are visually gated and require explicit review/approval in the UI; they are never ambient or one-click.
3. **Read canonical state, write canonical state.** The dashboard must not invent parallel fields. Empty/loading/error states come from the same state the CLI would expose with `--json`.
4. **Technical-local-first.** Dense tables, keyboard-friendly navigation, inline JSON/YAML snippets, and command-line copy helpers are features, not bugs.
5. **Progressive disclosure.** A new user sees a priority brief; drilling down reveals research, artifacts, evidence, diff, and approval queues.

## 2. Dashboard Screens / Sections

### 2.1 Global Shell (every screen)

A two-column shell:

- **Left rail (200 px, collapsible):** primary navigation.
- **Main area:** page content with a sticky page header containing the current profile selector and a global search/filter bar.

**Left rail nav (priority order):**

1. **Today / Priority Brief** — default landing page.
2. **Jobs** — discovery queue + full job list.
3. **Applications** — kanban and table tracker.
4. **Artifacts** — documents pending review/approval.
5. **Research** — company dossiers + stakeholder map.
6. **Outreach** — threads, drafts, due follow-ups.
7. **Tasks** — all tasks, not just application tasks.
8. **Profile & Proof** — profiles, resume source, answer bank.
9. **Automations** — runs, schedules, approval queue.
10. **Audit Log** — recent agent/user actions.

Each nav item shows a count badge only for actionable states: jobs requiring review, artifacts pending approval, tasks due today, follow-ups due, approvals pending, failed automation runs.

**Global header:**

- Active profile selector (dropdown of `Profile` records).
- "Sync / Refresh" button that re-reads state from the local API.
- "Command palette" shortcut (`Cmd/Ctrl + K`) to jump to any job, application, or artifact by ID or title/company.
- Local server status indicator (DB connected, last automation run).

---

### 2.2 Today / Priority Brief (default page)

**Purpose:** answer "what should I do right now?" for the active profile.

Sections, in order:

1. **Top 3 actions today**
   - Cards: `Apply to <Role> at <Company>`, `Review tailored resume for <Job>`, `Follow up with <Name>`.
   - Each card shows: entity link, deadline, why it matters (one-line), primary action button.
2. **Due today**
   - Mixed list of `Task`s, `OutreachThread` follow-ups, and `Application` deadlines that fall today or are overdue.
   - Sort: overdue first, then by time.
3. **New high-fit jobs**
   - Jobs with `status = new`, `fit_score >= profile.threshold`, imported since last user visit.
   - Each row: title/company, fit score pill, source, quick actions `Score details` / `Start research` / `Save`.
4. **Pending approvals**
   - Artifacts and staged external actions awaiting human approval.
5. **This week at a glance**
   - Mini timeline: interviews, follow-ups, application deadlines.

**Empty state:** "No actions for today. Run discovery or review your search preferences." with buttons to `Run discovery`, `Browse jobs`, `Edit profile`.

---

### 2.3 Jobs

Two sub-views toggled by tabs: **Queue** (default) and **All Jobs**.

#### Queue

- Saved filters: `New & unscored`, `Scored & unreviewed`, `High fit`, `Missing info`, `Saved for later`.
- Table columns:
  - Checkbox (for bulk actions: `Change status`, `Archive`, `Add to application`).
  - Company + title (link to job detail).
  - Source + posted date.
  - Fit score pill + confidence.
  - Status badge (`new`, `saved`, `researching`, `materials ready`, `ignored`).
  - Next action (derived from highest-priority open `Task` or missing artifact).
  - Quick actions: `View`, `Score`, `Research`, `Create application`.

#### Job Detail Page

A single-page view with these sections:

1. **Header card**
   - Title, company, location, work model, salary range, source URL.
   - Status selector + fit score + confidence.
   - Primary actions: `Start application`, `Generate dossier`, `Draft outreach`, `Archive`.
   - External-link icon opens source URL in a new tab; label: "Open posting (external)".
2. **Fit report panel**
   - Overall score, subscore bars (role fit, domain, seniority, comp, location, mission, growth, network, red flags).
   - Evidence bullets and missing-info warnings.
   - User override: "Edit rubric weights for this profile".
3. **Job description**
   - Collapsible, with keyword/tag chips extracted from requirements.
   - Toggle: "Show original" / "Show enriched".
4. **Company dossier** (if exists; otherwise prompt to generate)
   - Snapshot, red flags, recent news, interview angles.
5. **Stakeholder map**
   - List of `Stakeholder`s linked to this job/company.
   - Each row: name, inferred role, relevance tag, outreach status, quick action `Draft outreach`.
6. **Artifacts**
   - Tailored resume, cover letter, screening answers if any.
   - Show approval status, version, generated-by label.
7. **Application block** (collapsed until an `Application` exists)
   - Current stage, applied date, confirmation URL, notes, tasks, timeline.
8. **Activity / Audit log for this job**
   - Reverse-chronological list of agent/user events with command names and run IDs.

---

### 2.4 Applications

Two sub-views: **Kanban** and **Table**.

**Kanban columns** (mapped to `Application` stages):

`Saved` → `Researching` → `Materials Ready` → `Applied` → `Recruiter Screen` → `Interview` → `Offer` → `Rejected / Withdrawn / Ghosted`

- Each card shows: role @ company, fit score, next action, due date, artifact readiness indicators (resume ✓, cover letter ✓, answers ⚠).
- Drag-and-drop to change stage is optional; a dropdown stage changer is required.
- Clicking a card opens the same Job Detail page focused on the application block.

**Table columns:**

- Company / role.
- Stage badge.
- Applied date.
- Last activity.
- Next task + due date.
- Fit score.
- Source quality.
- Actions: `View`, `Add task`, `Update status`.

**Empty state:** "No applications yet. Save high-fit jobs from the Jobs queue to start tracking."

---

### 2.5 Artifacts

A review-centric list, not a file manager.

**Filters:** `Pending my review`, `Approved`, `Needs evidence`, `Generated by agent`, `Generated by me`.

**Table columns:**

- Artifact name + type (resume, cover letter, dossier, outreach draft, answers).
- Linked job / company / stakeholder.
- Version.
- Generated by (model/agent + run ID or "manual").
- Evidence status (pass / warn / fail).
- Approval status (draft / pending / approved / rejected).
- Actions: `Review`, `Diff`, `Approve`, `Reject`, `Export`.

**Artifact Review Page** (critical screen):

Two-pane layout:

- **Left: artifact render** (Markdown/HTML preview; PDFs open in a viewer panel).
- **Right: evidence & approval panel** with tabs:
  1. **Evidence** — list of claims mapped to proof points / job requirements. Flagged items shown first.
  2. **Diff** — if a previous version exists, highlight changes vs. master resume / prior tailored variant.
  3. **Prompt & config** — model, prompt version, profile used, job context (collapsible; for technical users).
  4. **Approval** — approve/reject with a required note if rejecting; optional note if approving.

Buttons at bottom:

- `Approve` (primary, disabled until user explicitly clicks).
- `Request changes` — sets status to `rejected` with note; creates a `Task` for the user or agent.
- `Export` — markdown / PDF / plain text.
- `Copy as CLI command` — e.g. `jobos artifacts approve artifact_456`.

---

### 2.6 Research

Two sub-views: **Companies** and **Stakeholders**.

#### Company dossier list

- Table: company, industry/stage, jobs linked, dossier freshness, red flags, actions `View dossier` / `Refresh`.
- Search by name or filter by `Has active application`.

#### Company dossier page

Sections:

1. Snapshot (product, market, business model, stage, funding, size).
2. Role/team context for linked jobs.
3. News & signals (date, source link, confidence).
4. Red flags with severity chips.
5. Interview angles & questions to ask.
6. Linked stakeholders.
7. Source citations (collapsible).

#### Stakeholder map

- Table: name, company, role, relationship to opening, outreach status, last contact.
- Filter by job, company, or `Due for follow-up`.
- Row actions: `View brief`, `Draft outreach`, `Mark do-not-contact`.

**Empty state:** "No research yet. Select a job and run `Generate dossier` or `Find stakeholders`."

---

### 2.7 Outreach

**Purpose:** manage `OutreachThread`s and follow-ups without accidentally sending anything.

Two sub-views: **Threads** and **Due follow-ups**.

#### Threads list

- Table: stakeholder, goal, channel, status, last message, next follow-up date, linked job.
- Actions: `View thread`, `Draft reply`, `Schedule follow-up`, `Pause thread`.

#### Thread detail page

- Header: stakeholder, goal, channel, relevance rationale, suppression state.
- Message history: drafts and sent messages clearly separated.
- For each draft:
  - `Quality score` (specificity, relevance, ask clarity, length, tone) with inline tips.
  - Evidence links (e.g. "mentions their blog post from 2024-03").
  - `Edit`, `Send` (gated), `Discard`, `Save to answer bank`.
- Follow-up scheduler: `after N business days`, pause conditions.

**External-action gate language for Send:**

- Button label: `Review & Send` (never just `Send`).
- On click: modal with:
  - Recipient, channel, job context.
  - Full message preview.
  - Checkbox: "I have reviewed this message and it is accurate."
  - Final button: `Send via <channel>`.
  - Post-send, log the event and update `OutreachThread` status.

**Empty state:** "No outreach threads. Research stakeholders from a job dossier first."

---

### 2.8 Tasks

- Unified task list across jobs, applications, outreach, and general admin.
- Columns: title, type, linked entity, due date, priority, status, created by.
- Bulk actions: mark done, snooze, change due date.
- Empty state: "No open tasks. The scheduler and agents create tasks as you save jobs and drafts."

---

### 2.9 Profile & Proof

Two sub-views: **Profiles** and **Resume Source / Answer Bank**.

#### Profiles

- List of `Profile` records with active indicator.
- Profile detail page:
  - Summary card: target roles, locations, salary band, constraints, communication style, risk tolerance.
  - Preference rubric with editable weights.
  - Proof points relevant to this profile.
  - Answer bank entries.
  - Calibration history.

#### Resume Source / Answer Bank

- Master resume sections in editable blocks.
- Proof points with evidence links.
- Answer bank: question → approved answer → usage count → linked applications.

**Empty state:** "Create a profile and import a resume to start. Try `jobos profile create --name 'PM EdTech'` from the CLI."

---

### 2.10 Automations

- **Schedules:** list of scheduled jobs, profile, frequency, last run, next run, external-side-effects flag.
- **Runs:** recent `AutomationRun`s with status, inputs, outputs, logs.
- **Approval queue:** any run that produced an artifact or staged external action needing approval.
- **Create automation:** simple form for the built-in schedules (`daily-discovery`, `followup-watch`, `interview-prep`, `stale-application-check`, `weekly-retrospective`).

**Empty state:** "No automations yet. Start with `daily-discovery` to populate your job queue each weekday morning."

---

### 2.11 Audit Log

- Reverse-chronological table: timestamp, actor (user / agent / model / scheduler), command/action, entity IDs, side effects (none / internal / external), success/failure.
- Filter by actor type, entity, or external-side-effects only.
- Export to JSONL.

## 3. Critical Empty, Loading, and Error States

### 3.1 Empty states (all required)

| Screen | Empty state message | Primary CTA |
|--------|---------------------|-------------|
| Today | "No actions for today. Run discovery or review your search preferences." | `Run discovery`, `Browse jobs` |
| Jobs | "No jobs found. Import a URL, run discovery, or paste a job description." | `Import job URL`, `Run discovery` |
| Applications | "No applications yet. Save high-fit jobs from the Jobs queue." | `Go to Jobs` |
| Artifacts | "No artifacts. Generate a tailored resume or cover letter from a job." | `Go to Jobs` |
| Research | "No research yet. Generate a dossier from a job." | `Generate dossier` |
| Outreach | "No outreach threads. Research stakeholders first." | `Go to Research` |
| Tasks | "No open tasks." | `Browse jobs` |
| Profile | "Create a profile and import a resume." | `Create profile` |
| Automations | "No automations yet." | `Add daily-discovery` |
| Audit Log | "No activity recorded yet." | `Run a command` |

**Design rule:** every empty state must offer a concrete next action that writes canonical state, not just a friendly illustration.

### 3.2 Loading states

- **Global refresh:** replace the main area with a skeleton loader and show "Reading local state…" in the header.
- **Table / list loading:** skeleton rows (5) with column headers.
- **Detail page loading:** show page title placeholder + three content skeleton blocks.
- **Long agent operations (e.g. generate dossier, tailor resume):** use a progress panel with status messages and a cancel button if the operation supports it. Show the `AutomationRun` ID.
- **Button loading:** primary button shows a spinner and disabled state; label becomes the active verb (e.g. "Generating…").

### 3.3 Error states

- **Local API unreachable:** full-page error with diagnostic copy: "Cannot reach JobOS backend at http://localhost:PORT. Is the server running?" Include a `Retry` button and the CLI command to start it (`jobos server start`).
- **No active profile:** persistent banner at top of every page: "No profile selected. Choose one in the header to score jobs and generate artifacts."
- **Command/agent failure:** inline alert card with run ID, command name, stderr snippet, and actions `View run log`, `Retry`, `Copy CLI command`.
- **Missing evidence on artifact:** block approval with a warning: "This artifact contains claims without source evidence. Review the Evidence tab."
- **External action blocked by policy:** explain clearly: "Auto-send is disabled by default. Go to Automations to change your policy, or use Review & Send to send manually."
- **Empty search/filter results:** show "No jobs match these filters" with a one-click `Clear filters`.

## 4. Artifact Review / Evidence Display

This is the most important UX surface for trust. Implement exactly this pattern on the Artifact Review Page:

### 4.1 Evidence panel

For every claim in the artifact, show a row:

- Claim text (truncated if long, expandable).
- Evidence tag:
  - `Directly supported` — green.
  - `Reasonably reframed` — blue.
  - `Needs user input` — yellow.
  - `Not allowed` — red.
- Source link: proof-point ID, resume section, job requirement, or research source.
- Confidence note.

### 4.2 Diff view

- If a tailored resume, show side-by-side: master resume section vs. tailored section.
- Highlight added, removed, reworded bullets.
- Flag any bullet whose meaning changed (not just wording).

### 4.3 Approval workflow

- Artifact statuses: `draft`, `pending_review`, `approved`, `rejected`.
- Only `approved` artifacts can be attached to an `Application` or used in `apply dry-run`.
- Rejecting an artifact creates a `Task` (`type: review`) linked to the artifact and the originating `AutomationRun`.
- Approving an artifact writes `approved_at`, `approved_by`, and an audit event.

## 5. External-Action Gate Language

Any action that could leave the local system must be visually and lexically distinct. Use this consistent language:

| Action type | Button label | Gate copy |
|-------------|--------------|-----------|
| Draft message | `Draft outreach` | none |
| Review draft | `Review & Send` | "This will send a message to <recipient> via <channel>. You cannot unsend it." |
| Apply dry-run | `Dry-run application` | "This will fill the form in a browser but will NOT submit it." |
| Submit application | `Review & Submit` | "This will submit your application to <company>. Confirm your artifacts are approved." |
| Run discovery | `Run discovery` | "This may query external job sources. Rate limits and source ToS apply." |
| Refresh dossier | `Refresh dossier` | "This will run web research and may call external search APIs." |
| Approve artifact | `Approve artifact` | "Approved artifacts may be used in applications." |

**Gate UI pattern:**

- External-action buttons are `outline` style until the user enters a review modal, where the final action button is `danger` or `primary` depending on severity.
- Disabled state with tooltip when prerequisites are missing (e.g. artifact not approved, profile missing).
- After any external action, show a confirmation with what happened, what did not happen, and a link to the audit log.

## 6. Navigation Priorities

### 6.1 Information architecture

```text
Today (default)
Jobs
  ├── Queue
  └── Job Detail
Applications
  ├── Kanban
  ├── Table
  └── Job Detail (application focus)
Artifacts
  ├── List
  └── Review
Research
  ├── Companies
  ├── Company Dossier
  └── Stakeholders
Outreach
  ├── Threads
  ├── Due Follow-ups
  └── Thread Detail
Tasks
Profile & Proof
  ├── Profiles
  └── Resume Source / Answer Bank
Automations
  ├── Schedules
  ├── Runs
  └── Approval Queue
Audit Log
```

### 6.2 Keyboard shortcuts

- `?` — show shortcut help.
- `/` or `Cmd/Ctrl + K` — open command palette.
- `g` then `j` — go to Jobs.
- `g` then `a` — go to Applications.
- `g` then `r` — go to Artifacts.
- `g` then `t` — go to Tasks.
- `n` then `j` — new job import.
- `Escape` — close modals / panels.

### 6.3 Mobile / small viewport

- Collapse left rail to a bottom tab bar with the five highest-priority items: Today, Jobs, Applications, Artifacts, Tasks.
- Hide secondary screens behind a "More" menu.
- Tables become stacked cards.

## 7. Visual and Accessibility Checks for MVP

### 7.1 Visual checks

- **Color semantics:**
  - Fit score: low = gray/red, medium = yellow, high = green. Do not rely on color alone; show the numeric score.
  - Status badges: consistent with `Application` stages.
  - External-action buttons: `danger`/`warning` style, never plain links.
- **Density:** default to compact tables; allow a "comfortable" toggle.
- **Monospace for IDs and CLI commands:** every ID (`job_123`, `artifact_456`, `run_789`) is rendered in monospace and selectable.
- **CLI copy helpers:** next to any dashboard action, show a small copy icon that copies the equivalent CLI command (e.g. `jobos applications create --job job_123 --profile pm-edtech`).

### 7.2 Accessibility checks

- All navigation is reachable by keyboard; focus indicators visible.
- Tables have proper `scope` and sort headers.
- Status badges and score pills have text equivalents (e.g. `aria-label="Fit score 4.2 out of 5"`).
- Modals trap focus; `Escape` closes.
- Color contrast minimum 4.5:1 for text; do not use red/green as the only signal.
- Loading states announced via `aria-live="polite"`.
- External-action confirmation modals use `aria-live="assertive"` for the warning sentence.
- Every button's label matches its consequence (no "OK" / "Click here").

## 8. Data Contract for the Single HTML Dashboard

The dashboard is a single-page application (or a small multi-page app) served from the local JobOS server. It must read from and write to the canonical REST/OpenAPI surface, not a bespoke dashboard API.

### Required endpoints for MVP

| Screen | Endpoint(s) |
|--------|-------------|
| Today | `GET /api/today?profile={id}` (composite) |
| Jobs list | `GET /api/jobs?status=&min_fit=&sort=` |
| Job detail | `GET /api/jobs/{id}` (with dossier, stakeholders, artifacts, application, audit embedded or linked) |
| Update job status | `PATCH /api/jobs/{id}` |
| Applications kanban/table | `GET /api/applications?profile=` |
| Update application stage | `PATCH /api/applications/{id}` |
| Artifacts list | `GET /api/artifacts?status=` |
| Artifact detail | `GET /api/artifacts/{id}` |
| Approve/reject artifact | `POST /api/artifacts/{id}/approve`, `/reject` |
| Research | `GET /api/companies`, `GET /api/stakeholders` |
| Dossier detail | `GET /api/companies/{id}` |
| Outreach | `GET /api/outreach`, `GET /api/outreach/{id}` |
| Send outreach | `POST /api/outreach/{id}/send` (gated) |
| Tasks | `GET /api/tasks`, `PATCH /api/tasks/{id}` |
| Profiles | `GET /api/profiles`, `GET /api/profiles/{id}` |
| Resume source | `GET /api/resume-source`, `PATCH /api/resume-source` |
| Automations | `GET /api/automations`, `POST /api/automations`, `GET /api/automation-runs` |
| Audit log | `GET /api/audit-log` |

### State update rules

- The dashboard must poll or use server-sent events to reflect CLI/agent changes without a full page reload.
- When the user performs an action in the UI, prefer issuing the same REST call an agent would use, then re-fetching the affected entity.
- Never maintain a separate dashboard-only cache that diverges from canonical state.

## 9. Acceptance Checklist for the Handoff

This handoff is complete enough to implement a single local HTML dashboard when the following can be answered concretely:

- [ ] The dashboard reads from the same REST/OpenAPI endpoints the CLI and agents use.
- [ ] Every primary screen in Section 2 has a defined layout, columns, and primary CTA.
- [ ] Empty, loading, and error states from Section 3 are specified for every primary screen.
- [ ] Artifact review includes evidence tags, diff view, prompt/config disclosure, and explicit approve/reject flow.
- [ ] External actions use the gate language and UI pattern from Section 5 and cannot be triggered accidentally.
- [ ] Navigation priorities and keyboard shortcuts from Section 6 are defined.
- [ ] Visual and accessibility checks from Section 7 are explicit enough to test.
- [ ] The data contract in Section 8 lists the minimum endpoints required.

---

## 10. What to Defer (out of scope for this handoff)

- Browser extension UX or any job-page capture flow.
- External sending/applying UI beyond the review-and-send gate described here.
- Consumer SaaS polish: onboarding animations, empty-state illustrations, paid-tier prompts, email notifications.
- Multi-user auth, roles, or tenant switching.
- Advanced analytics dashboards (funnel charts, A/B variant reporting).
- Mobile-native app design; only responsive web is covered.
